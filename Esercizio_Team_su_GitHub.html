Esercizio: Team su GitHub

Metti alla prova le conoscenze acquisite su collaboratori, branch e fork. Risponde alle domande e completa l'esercizio pratico guidato.

~20 minuti 10 domande + esercizio pratico

1 Collaboratori

Verifica la comprensione del meccanismo dei collaboratori su GitHub: accessi, ruoli e best practice.

1 Cosa si intende per collaboratore su GitHub? 1 pt
  • Un utente che ha effettuato il fork del repository
  • Un utente a cui il proprietario ha concesso accesso diretto in scrittura al repository
  • Qualsiasi utente che ha aperto una Issue sul repository
  • Un utente che ha messo una stella (⭐) al repository
2 Quale percorso bisogna seguire su GitHub per aggiungere un collaboratore? 1 pt
  • Profile → Developer Settings → Collaborators
  • Repository → Insights → Contributors
  • Repository → Settings → Collaborators and teams → Add people
  • Repository → Pull requests → Reviewers
3 Metti in ordine corretto i passaggi per aggiungere un collaboratore (trascina i blocchi) 1 pt
  • ⠿Scegliere il livello di accesso (Read, Write, Admin…)
  • ⠿Aprire le impostazioni del repository (Settings)
  • ⠿Il collaboratore accetta l'invito via email entro 7 giorni
  • ⠿Cliccare su "Collaborators and teams" → "Add people" e cercare l'utente
4 Quale affermazione sui ruoli di collaboratore è corretta? 1 pt
  • Esiste un unico livello di accesso per tutti i collaboratori: write
  • Un collaboratore con ruolo Admin può eliminare branch protetti e modificare le impostazioni del repository
  • Il ruolo Read permette di fare push diretto sul branch main
  • I collaboratori non possono aprire Pull Request

2 Branch

Domande sui branch: cosa sono, come funzionano e quando usarli.

5 Nel diagramma qui sopra, il commit C? è il terzo commit sul branch feature. Dopo il merge nel main, cosa succede al branch feature? 1 pt
  • Viene eliminato automaticamente da GitHub
  • Rimane disponibile; va eliminato manualmente (o cliccando "Delete branch" dopo la PR)
  • I suoi commit vengono cancellati dalla storia del repository
  • Diventa automaticamente il nuovo branch principale
6 Quale affermazione descrive correttamente un branch in Git? 1 pt
  • Creare un branch duplica fisicamente tutti i file del repository
  • Un branch è un puntatore mobile a uno specifico commit; la creazione è istantanea e leggera
  • Un repository può avere al massimo 3 branch attivi contemporaneamente
  • I branch sono disponibili solo nei repository pubblici
7 Completa la frase: "Un branch con vita molto lunga tende ad accumulare _______ difficili da risolvere." 1 pt

3 Fork

Domande sul fork: differenze con i branch, casi d'uso e flusso contributo.

8 Qual è la differenza principale tra un fork e un branch? 1 pt
  • Il fork crea una copia indipendente del repository nell'account di un altro utente; il branch è una linea di sviluppo interna allo stesso repository
  • Non c'è differenza: fork e branch sono sinonimi su GitHub
  • Il fork è disponibile solo ai collaboratori con ruolo Admin
  • Il branch crea un repository separato, il fork crea un ramo nello stesso repository
9 Quando si usa tipicamente un fork piuttosto che diventare collaboratore diretto? 1 pt
  • Quando si vuole contribuire a un repository open source su cui non si ha write access diretto
  • Quando si vuole correggere un bug urgente sul branch main del proprio team
  • Quando si vuole aggiungere un collaboratore al proprio repository
  • Quando si vuole proteggere il branch main con delle regole
10 Scenario: vuoi correggere un errore nella documentazione di un progetto open source che non è tuo. Qual è il flusso corretto? 1 pt
  • Aprire una Issue → attendere che il maintainer sistemi il problema
  • Fork del repository → creare un branch nella propria copia → apportare le modifiche → aprire una Pull Request verso il repository originale
  • Clonare il repository originale → fare push direttamente sul main dell'originale
  • Contattare il proprietario via email chiedendo di aggiungere la modifica

4 Esercizio Pratico

Segui le istruzioni passo dopo passo e svolgile direttamente su GitHub. Questo esercizio simula un flusso reale di collaborazione in un progetto scolastico.

ℹ️
Account GitHub dei collaboratori

In questo esercizio utilizzerai i seguenti account GitHub come collaboratori del tuo repository: frandelfo   francescoadriani

🏗️ Task A — Crea il repository e aggiungi i collaboratori Facile

Crea un nuovo repository pubblico chiamato progetto-team sul tuo account GitHub personale, poi invita entrambi i collaboratori indicati con ruolo Write.

1

Crea il repository

Vai su github.com → clicca il + in alto a destra → New repository. Assegna il nome progetto-team, impostalo come Public e spunta "Add a README file". Clicca Create repository.

2

Aggiungi il primo collaboratore

Dal repository appena creato vai in Settings → Collaborators and teams → Add people. Cerca l'utente frandelfo e seleziona il ruolo Write.

3

Aggiungi il secondo collaboratore

Ripeti il passo precedente cercando l'utente francescoadriani, sempre con ruolo Write.

4

Verifica gli inviti

Controlla che nella sezione Collaborators and teams appaiano entrambi gli utenti con lo stato "Invited" o "Active" (dipende se hanno già accettato).

✅

Come verificare il completamento: nella pagina Settings → Collaborators and teams devono apparire entrambi gli account elencati sopra.

🌿 Task B — Crea un branch e aggiungi un file Medio

Senza lasciare GitHub.com (usa l'interfaccia web), crea un branch chiamato sviluppo a partire dal main, poi aggiungi un nuovo file al progetto su quel branch.

1

Crea il branch sviluppo

Nella pagina principale del repository clicca sul selettore branch (mostra main → scorciatoia). Digita sviluppo nel campo di ricerca e scegli "Create branch: sviluppo from main".

2

Crea un nuovo file sul branch sviluppo

Assicurati di essere sul branch sviluppo (il selettore in alto deve mostrare "sviluppo"). Clicca Add file → Create new file. Chiama il file contributo.md e scrivi al suo interno almeno una riga di testo, ad esempio:
# Il mio contributo
Poi clicca Commit changes… e scegli "Commit directly to the sviluppo branch".

3

Apri una Pull Request

GitHub mostrerà un banner giallo "Compare & pull request". Cliccalo. Inserisci come titolo "Aggiunto file contributo" e clicca Create pull request. Non fare ancora il merge.

⚠️

Non fare il merge della Pull Request prima che il docente l'abbia verificata. Il merge è l'ultima operazione da eseguire solo dopo l'approvazione.

🔁 Task C — Simula un fork (opzionale, livello avanzato) Avanzato

Utilizza il profilo di uno dei due collaboratori (o un secondo account personale) per eseguire un fork del repository progetto-team, apportare una modifica e inviare una Pull Request verso il repository originale.

1

Fai il fork

Accedi con l'account francescoadriani (o frandelfo). Naviga fino al repository progetto-team. In alto a destra clicca Fork → Create fork.

2

Modifica il README nel fork

Nel tuo fork, apri il file README.md, clicca l'icona matita ✏️ e aggiungi una riga con il nome del collaboratore. Fai il commit sul branch main del fork.

3

Apri una PR verso l'originale

Nel fork comparirà il banner "Contribute → Open pull request". Cliccalo, inserisci un titolo descrittivo e clicca Create pull request. La PR arriverà al repository progetto-team originale.

4

Accetta o rifiuta la PR (come owner)

Torna al tuo account originale (proprietario del repository). Apri la Pull Request ricevuta, lascia un commento e poi decide se fare il merge o chiuderla.

📬 Consegna il tuo lavoro

Al termine dei Task A, B e C copia il link al tuo repository GitHub e incollalo qui sotto, poi clicca Salva link. Consegna questo link al docente per la verifica del completamento della parte pratica.

Esempio: https://github.com/tuo-username/progetto-team

✅ Link salvato! Mostra questa pagina al docente oppure copia il link e invialo tramite il canale indicato.

5 Esercizio con GitHub Desktop

In questo esercizio utilizzerai GitHub Desktop per clonare il repository, creare un branch locale, fare modifiche e sincronizzare tutto con GitHub, senza usare il terminale né l'interfaccia web per le operazioni Git.

⚠️
Prerequisito

Assicurati di aver completato il Task A (repository progetto-team già creato e collaboratori aggiunti) prima di iniziare questo esercizio. GitHub Desktop deve essere installato e configurato con il tuo account GitHub.

🖥️ Task D — Clona, modifica e sincronizza con GitHub Desktop Medio
🖥️ GitHub Desktop — app per Windows e macOS

Utilizzerai GitHub Desktop per clonare il repository progetto-team sul tuo computer, creare un branch chiamato desktop-edit, modificare un file localmente e pubblicare le modifiche su GitHub aprendo una Pull Request.

1

Clona il repository con GitHub Desktop

Apri GitHub Desktop. Nel menu in alto scegli File → Clone repository…. Nella scheda GitHub.com cerca il repository progetto-team e scegli una cartella locale dove salvarlo. Clicca Clone.

File → Clone repository… → GitHub.com → progetto-team → Clone
2

Crea un nuovo branch locale

In GitHub Desktop, clicca sul menu Current Branch (barra in alto al centro). Si apre un pannello: clicca New Branch, digita il nome desktop-edit e clicca Create Branch. Ora sei sul branch desktop-edit.

Current Branch ▾ → New Branch → desktop-edit → Create Branch
3

Modifica un file con un editor di testo

Clicca Open in… (o usa il menu Repository → Open in Explorer / Show in Finder) per aprire la cartella del repository. Apri il file README.md con Blocco Note, TextEdit o VS Code. Aggiungi in fondo una riga, ad esempio:

## Modificato con GitHub Desktop

Salva il file.

4

Fai il commit con GitHub Desktop

Torna in GitHub Desktop: nella colonna di sinistra vedrai il file README.md evidenziato come modificato. In basso a sinistra compila il campo Summary con il messaggio Aggiunta sezione desktop, poi clicca Commit to desktop-edit.

Summary: "Aggiunta sezione desktop" → Commit to desktop-edit
5

Pubblica il branch su GitHub

In alto comparirà il pulsante Publish branch. Cliccalo: GitHub Desktop invierà (push) il branch desktop-edit sul repository remoto.

Publish branch
6

Apri la Pull Request su GitHub.com

Dopo il push, GitHub Desktop mostra il pulsante Create Pull Request. Cliccalo: si aprirà il browser direttamente sulla pagina di creazione PR su GitHub.com. Inserisci il titolo "Modifica da GitHub Desktop" e clicca Create pull request. Non fare ancora il merge.

Create Pull Request → GitHub.com si apre automaticamente → Create pull request
✅

Come verificare il completamento: sul repository progetto-team deve essere visibile una Pull Request aperta con titolo "Modifica da GitHub Desktop" proveniente dal branch desktop-edit.

📬 Consegna — Esercizio GitHub Desktop

Copia il link diretto alla Pull Request aperta (es. https://github.com/username/progetto-team/pull/2) e incollalo qui sotto per consegnare anche questa parte al docente.

✅ Link PR salvato! Mostra questa pagina al docente oppure invia il link tramite il canale indicato.
